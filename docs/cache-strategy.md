# 缓存策略详解

## 核心原则

基于 Anthropic Claude Code 的 prompt caching 机制：
- **前缀匹配** — API 缓存从 byte 0 到 `cache_control` 断点
- **任何前缀变更 = 缓存全部失效**
- **5 分钟 TTL**（订阅版可 1 小时）
- **静态在前、动态在后**

## 我们的策略

### 1. 前缀结构

```
┌─────────────────────────────────────────────┐
│ [系统 prompt + 工具定义]     ← 全局缓存      │
│ [CLAUDE.md 内容]             ← 项目级缓存    │
├─────────────────────────────────────────────┤ ← cache_control 断点
│ [memory-index.md 静态部分]   ← 固定结构      │
│   - 缓存锚点版本号            ← 永远不变      │
│   - 当前状态（1 句）          ← 可追加        │
│   - 活跃决策列表              ← 可追加        │
├─────────────────────────────────────────────┤ ← 动态边界
│ [daily/ 近3天摘要]           ← 新日替换旧日  │
│ [用户消息]                   ← 消息层        │
└─────────────────────────────────────────────┘
```

### 2. 硬约束（违反即缓存失效）

| 规则 | 原因 |
|------|------|
| 禁止会话中编辑 memory-index.md | 前缀变化 = 全部失效 |
| 禁止会话中编辑 CLAUDE.md | 同上 |
| memory-index.md 只追加 | 追加不改变已有前缀 |
| daily/ 只在 Stop 时写入 | 保护对话中缓存 |
| 不切换模型 | 模型隔离缓存 |

### 3. 缓存锚点机制

memory-index.md 中包含不可变的版本标记：
```markdown
<!-- CACHE_ANCHOR: memory-index-v1 — DO NOT MODIFY THIS LINE -->
```

- SessionStart 时验证锚点完整性
- 维护扫描时检查锚点
- 如果锚点被修改 → CI 失败 → 人工修复

### 4. TTL 保活策略

- 交互频繁场景：自然保持 5 分钟 TTL
- 长暂停后重启：使用 `/new` 而非恢复旧会话
- 重度任务：考虑保持对话活跃避免 TTL 过期

### 5. 监控指标

| 指标 | 健康阈值 | 告警阈值 |
|------|----------|----------|
| 缓存命中率 | > 90% | < 85% |
| index tokens | < 800 | > 1000 |
| 前缀变更次数 | 0 | > 0 |
| 活跃记忆数 | < 15 | > 20 |
